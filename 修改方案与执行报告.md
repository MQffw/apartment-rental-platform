## 一、消息通知系统改造

### 1.1 现状问题

- 前端调用 `GET /app/notification/list`，后端只有 `page` 分页接口，接口不匹配
- `sendNotification()` 方法已定义但全项目无任何调用，通知从未被创建
- `notification` 表无索引，`user_id` 和 `is_read` 查询全表扫描
- 存在两个重复的通知页面（`message.vue` 和 `notification.vue`）

### 1.2 通知类型与触发场景

| type | 名称 | 触发场景 | 生产端 | 通知内容示例 |
|------|------|---------|--------|------------|
| 1 | 租约到期提醒 | 定时任务每天8:00 | web-app ScheduledTask | "您在XX公寓XX房间的租约将于7天后到期" |
| 2 | 预约状态变更 | 管理员变更预约状态 | web-admin AppointmentService | "您的预约状态已变更为：已确认" |
| 3 | 新预约通知 | 用户提交看房预约 | web-app AppointmentService | "预约看房已提交，请等待确认" |
| 4 | 租约状态变更 | 签约审批/退租确认等 | web-admin/web-app AgreementService | "您的租约已审批通过" |
| 5 | 缴费提醒 | 每月出账/逾期提醒 | web-app ScheduledTask | "您本月房租 ¥1500 已出账，请及时支付" |
| 6 | 帖子被赞 | 有人点赞你的帖子 | web-app PostLikeConsumer | "XXX 赞了你的评价「XX公寓住后感」" |
| 7 | 帖子被评论 | 有人评论/回复你的帖子 | web-app PostCommentConsumer | "XXX 评论了你的评价：「采光确实很好」" |

### 1.3 RabbitMQ 异步投递架构

所有通知统一走 MQ，业务接口不直接写 notification 表。

```
                        ┌──────────────────┐
                        │   notification    │
                        │    .exchange      │
                        │  (Direct类型)      │
                        └────────┬─────────┘
                                 │ 路由键: notification.send
                                 ▼
┌──────────┐    发消息    ┌──────────────────┐   消费    ┌─────────────────┐
│ 业务Service ├──────────>│ notification     ├─────────>│ Notification    │──> 写DB
│ (admin/app)│           │   .queue         │          │   Consumer      │
└──────────┘    发消息    │ (持久化+手动ACK)   │          │ (手动ACK)       │──> INCR Redis未读数
                        └────────┬─────────┘          └─────────────────┘
                                 │ 消费失败3次
                                 ▼
                        ┌──────────────────┐
                        │ notification     │
                        │   .dead.queue    │
                        │  (死信队列兜底)    │
                        └──────────────────┘
```

**消息体 DTO**：

```java
public class NotificationMessage {
    Long userId;           // 接收用户ID
    String title;          // 通知标题
    String content;        // 通知内容
    Integer type;          // 通知类型 1-7
    Long relatedId;        // 关联业务ID（租约ID/帖子ID等）
}
```

**RabbitMQ 队列声明（共用一个 RabbitConfig，放在 common 模块）**：

| 名称 | 类型 | 说明 |
|------|------|------|
| `notification.exchange` | Direct | 通知交换机 |
| `notification.queue` | Queue | 主队列，持久化 |
| `notification.dead.exchange` | Direct | 死信交换机 |
| `notification.dead.queue` | Queue | 死信队列，消费失败兜底 |

### 1.4 Redis 缓存未读数

- Key：`notification:unread:{userId}`，Value：Integer
- 消费者写通知时 `INCR`
- 用户标记全部已读时 `DEL`
- 用户标记单条已读时 `DECR`（最小为 0，用 Lua 保证）
- 前端 `/unreadCount` 接口先查 Redis，未命中再查 DB 回填

### 1.5 修复与优化

- 后端新增 `GET /app/notification/list` 接口（返回最近 50 条，不分页）
- 添加索引：`ALTER TABLE notification ADD INDEX idx_user_read (user_id, is_read)`
- 删除重复的 `notification.vue` 页面，统一用 `message.vue`

## 三、圈子 → 高并发点评社区

### 3.1 定位

将现有圈子（纯静态 mock）改造为**公寓点评社区**，用户可以对公寓/房间发布评价、打分、上传图片，其他用户可以点赞、评论。重点体现高并发场景下的技术方案。

### 3.2 高并发场景与解决方案

| 场景 | 并发特征 | 解决方案 | 中间件 |
|------|---------|---------|--------|
| 帖子点赞 | 同一帖子短时间大量点赞，写冲突 | Redis Lua 原子操作 + RabbitMQ 异步落库 | Redis + RabbitMQ |
| 帖子评论 | 热门帖子评论刷屏，高并发写 | RabbitMQ 异步入库 + Redis 计数缓存 | Redis + RabbitMQ |
| 帖子列表 | 读多写少，热门帖子反复访问 | Redis 缓存 + 分布式锁防击穿 | Redis |
| 发布帖子 | 写操作含图片，瞬时并发 | RabbitMQ 异步写入，接口秒返回 | RabbitMQ |
| 防重复点赞 | 同用户同帖子只能赞一次 | Redis Set + Lua 原子判断 | Redis |
| 接口限流 | 防刷帖/刷赞/刷评论 | Redis + Lua 滑动窗口 | Redis |
| 热门排行 | 按热度/点赞排序 | Redis ZSet 维护 Top100 | Redis |

### 3.3 数据库设计

```sql
-- 帖子表

    rating        TINYINT COMMENT '评分1-5',
    like_count    INT DEFAULT 0 COMMENT '点赞数（定时从Redis回写）',
    comment_count INT DEFAULT 0 COMMENT '评论数（定时从Redis回写）',
    status        TINYINT DEFAULT 1 COMMENT '1-正常 2-审核中 3-已屏蔽',
    create_time   DATETIME,
    update_time   DATETIME,
    is_deleted    TINYINT DEFAULT 0,
    INDEX idx_user (user_id),
    INDEX idx_apartment (apartment_id),
    INDEX idx_create (create_time),

-- 评论表

    parent_id    BIGINT DEFAULT 0 COMMENT '父评论ID（0=一级）',
    reply_to_id  BIGINT DEFAULT 0 COMMENT '回复目标用户ID',

-- 点赞持久化表（日常走Redis，定时批量回写）
CREATE TABLE post_like (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
```

### 3.4 RabbitMQ 队列设计（3 条队列）

| 队列名 | 用途 | 消息体 | 消费者动作 |
|--------|------|--------|-----------|
| `post.create.queue` | 发帖异步入库 | PostCreateMsg{id,userId,title,content,images,rating,apartmentId} | INSERT post_info + 写Redis缓存 |
| `post.like.queue` | 点赞异步落库 | PostLikeMsg{postId,userId,action:"like"/"unlike"} | INSERT/DELETE post_like |
| `post.comment.queue` | 评论异步入库 | PostCommentMsg{postId,userId,parentId,replyToId,content} | INSERT post_comment + INCR comment_count |

所有队列配死信队列兜底。交换机统一用 Direct 类型。

### 3.5 核心流程详解

#### 3.5.1 点赞流程（Redis Lua + RabbitMQ）

```
用户点击"赞"
    │
    ▼
Redis Lua 原子脚本:
    KEYS[1] = post:like:{postId}:users  (Set)
    KEYS[2] = post:like:{postId}:count  (String)

    local isMember = redis.call('SISMEMBER', KEYS[1], userId)
    if isMember == 1 then
        redis.call('SREM', KEYS[1], userId)
        redis.call('DECR', KEYS[2])
        return {0, redis.call('GET', KEYS[2])}   -- 取消赞, 新count
    else
        redis.call('SADD', KEYS[1], userId)
        redis.call('INCR', KEYS[2])
        return {1, redis.call('GET', KEYS[2])}   -- 赞, 新count
    end
    │
    ▼
前端立即拿到结果，更新UI（无延迟感）
    │
    ▼
发送 MQ → post.like.queue
    消息: {postId, userId, action: "like"/"unlike"}
    │
    ▼
消费者 PostLikeConsumer:
    like  → INSERT IGNORE INTO post_like
    unlike → DELETE FROM post_like
    │
    ▼
点赞成功后，消费者同时发通知消息 → notification.queue
    通知帖子作者: "XXX 赞了你的评价「XX公寓住后感」"
```

**定时回写**：`@Scheduled(fixedRate=300000)` 每 5 分钟扫描所有 Redis `post:like:*:count`，与 DB 比对差异，批量 UPDATE post_info.like_count。

#### 3.5.2 评论流程（RabbitMQ 异步写入）

```
用户提交评论
    │
    ▼
Controller: 限流检查（Redis Lua 滑动窗口）
    │ 限流规则: 同一用户 10 秒内最多评论 3 次
    │
    ▼
生成评论ID（Redis INCR），立即返回 commentId
    │
    ▼
发送 MQ → post.comment.queue
    消息: {commentId, postId, userId, parentId, replyToId, content}
    │
    ▼
消费者 PostCommentConsumer:
    1. INSERT INTO post_comment
    2. Redis INCR post:comment:{postId}:count
    3. 发通知消息 → notification.queue
       通知帖子作者: "XXX 评论了你的评价：「确实不错」"
```

#### 3.5.3 发帖流程（RabbitMQ 异步写入）

```
用户提交帖子
    │
    ▼
Controller: 参数校验 + 限流检查
    │ 限流规则: 同一用户 60 秒内最多发 1 篇
    │
    ▼
生成帖子ID（Redis INCR），立即返回 postId
    │
    ▼
发送 MQ → post.create.queue
    消息: {postId, userId, title, content, images, rating, apartmentId}
    │
    ▼
消费者 PostCreateConsumer:
    1. INSERT INTO post_info
    2. 写入 Redis 缓存 post:detail:{postId}
    3. 如果关联了公寓，更新公寓的平均评分
```

#### 3.5.4 帖子列表缓存（防穿透/击穿）

```
请求帖子列表
    │
    ▼
查 Redis: key = post:list:{page}:{sort}
    │
    ├── 命中 → 返回
    │
    └── 未命中 → 加分布式锁 lock:post:list:{page}
                  │
                  ├── 获取成功 → 查DB → 写缓存(TTL 5min) → 释放锁
                  └── 获取失败 → 等待50ms → 重查缓存

防穿透: 空结果也缓存（TTL 60s）
```

#### 3.5.5 限流（Redis + Lua 滑动窗口）

```lua
-- KEYS[1] = rate:limit:{userId}:{action}
-- ARGV[1] = 限流次数, ARGV[2] = 时间窗口(秒)
local current = redis.call('GET', KEYS[1])
if current and tonumber(current) >= tonumber(ARGV[1]) then
    return 0  -- 拒绝
end
redis.call('INCR', KEYS[1])
if tonumber(redis.call('TTL', KEYS[1])) == -1 then
    redis.call('EXPIRE', KEYS[1], ARGV[2])
end
return 1  -- 放行
```

| 场景 | 限流 Key | 规则 |
|------|---------|------|
| 发帖 | `rate:limit:{userId}:post` | 60 秒 1 次 |
| 点赞 | `rate:limit:{userId}:like` | 1 秒 10 次 |
| 评论 | `rate:limit:{userId}:comment` | 10 秒 3 次 |

#### 3.5.6 热门排行（Redis ZSet）

```
Key: post:hot:rank
Score: likeCount * 10 + commentCount * 5 - 时间衰减(小时数 * 2)

@Scheduled(cron = "0 0 * * * ?")  每小时更新:
    1. 查最近7天帖子
    2. 算热度分
    3. ZADD post:hot:rank score postId
    4. ZREMRANGEBYRANK 保留Top100

前端请求:
    ZREVRANGE post:hot:rank 0 19  →  Top20帖子ID
    Pipeline MGET post:detail:{id}  →  批量取详情
```

## 五、技术栈总结（简历亮点）

| 技术点 | 应用场景 | 体现能力 |
|-------|---------|---------|
| RabbitMQ | 通知投递/发帖落库/点赞落库/评论落库 | 消息解耦、异步处理、最终一致性 |
| 死信队列 | 消费失败兜底 | 消息可靠性保障 |
| Redis Lua | 点赞原子操作/限流滑动窗口 | 并发安全、原子性保证 |
| Redis Set/ZSet | 点赞去重/热门排行 | 数据结构选型 |
| Redis 分布式锁 | 缓存击穿防护 | 分布式并发控制 |
| Redis INCR+定时回写 | 点赞数/评论数高并发 | 读写分离、批量优化 |
| 防缓存穿透 | 空值缓存 | 缓存高可用 |
| 滑动窗口限流 | 防刷帖/刷赞/刷评论 | 系统保护、防滥用 |
| 定时任务 | 租约到期提醒/热度更新/计数回写 | 调度与批处理 |
| CAS状态机 | 租约状态并发修改（已有） | 并发安全 |
| 管理后台联动 | 报修管理/帖子审核/公告发送 | 全栈业务闭环 |
